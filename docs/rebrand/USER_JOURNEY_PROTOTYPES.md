# User Journey Prototypes

These are documentation prototypes: sequence, decisions, evidence and recovery are testable before visual styling. Every journey can reach the scan explanation in three meaningful choices or fewer from its landing page.

## Journey A — Operations-led visitor

- **Trigger/landing:** Manual handoffs, spreadsheet dependence, late errors or scaling pain; lands on homepage or Workflow modernization via referral/search.
- **Questions:** Is the problem recognizable? Will exceptions/human decisions survive? Must we replace systems? What disruption and internal input are involved?
- **Objections:** “Automation will ignore reality”; “IT already has too much work”; “a scan is paid presales”; “we cannot expose sensitive data.”
- **Content consumed:** Conflict → Hidden Drag → workflow route → fit/non-fit → illustrative before/after with human control → scan outputs/limits → founder/method proof.
- **Decision points:** Recognize ≥2 friction signals? Choose workflow or combined? Is the standalone scan output useful? Ready for intake?
- **Likely exits:** Workflow case, Method, email/share with IT, or abandonment at unclear scan facts.
- **Conversion:** `Start een workflow-led scan` → route preselected → intake → confirmed receipt.
- **Fallback:** `Plan een verkennend gesprek`/email with warning not to send confidential system data.
- **Required proof:** One evidence-labelled workflow case, current-state map excerpt, exception/human-control example, verified handover artifact.

## Journey B — IT-led visitor

- **Trigger/landing:** Unsupported dependencies, fragile integration, maintenance/security concern or missing tests/docs; lands on application service or technical insight.
- **Questions:** Is a rewrite assumed? How deep is the assessment? What about auth, data, integrations, testing, migration and rollback? Can a founder-led studio handle scope safely?
- **Objections:** Marketing-level technical language; vague AI claims; security guarantees; black-box delivery; uncontrolled access requests.
- **Content consumed:** Application warning signs/layers → assessment boundaries → decision options → phased migration/quality gates → relevant case → Method/About → application scan.
- **Decision points:** Is issue application-led or combined? Are limits honest? Is evidence sufficiently technical? Is public intake safe?
- **Likely exits:** Share insight/case internally, inspect About/Method, contact directly for NDA/access process.
- **Conversion:** `Start een application-led scan` → document/access availability question without upload → intake.
- **Fallback:** Email/contact for procurement or confidentiality constraints.
- **Required proof:** Application case with dependency/architecture evidence, option rationale, tests/migration/rollback artifact and explicit security-assessment boundary.

## Journey C — Founder/product-led visitor

- **Trigger/landing:** MVP/internal tool has outgrown prototype quality; lands on insight, Application modernization or homepage.
- **Questions:** Refactor, replatform or rebuild? Can current product value be preserved? How will scope and ownership be transferred? Is this a fractional CTO offer?
- **Objections:** Fear of total rewrite, open-ended consulting, agency handoffs or generic stack sales.
- **Content consumed:** Relevant insight → application options → Method → founder capability/boundaries → case → scan.
- **Decision points:** Is core pain product/application or combined workflow? Does staged route preserve business continuity? Does direct founder involvement fit?
- **Likely exits:** Save/share insight, contact for fit, or leave if case relevance is weak.
- **Conversion:** Application/combined scan with desired outcome and timing.
- **Fallback:** Verkennend gesprek for product context, explicitly not a free technical audit.
- **Required proof:** Modernization decision case, verified founder chronology, decision log/roadmap sample and transferability evidence.

## Journey D — Credibility validation

- **Trigger/landing:** Referral, prior visit or proposal validation; lands on homepage, Cases or About.
- **Questions:** Is this one person or an agency? What is proven? Are claims current and permitted? How is delivery governed? What happens when specialists are needed?
- **Objections:** Thin/fabricated proof, inflated experience, anonymous lab work presented as clients, broad tool lists, unclear accountability.
- **Content consumed:** Evidence policy → case classification/detail → limitations/not built → Method artifacts → About verified facts → scan/contact.
- **Decision points:** Does evidence type match expectation? Are claims traceable? Are scope and partner boundaries credible? Is next step proportionate?
- **Likely exits:** Download/share case, direct contact, or rejection due to unresolved proof.
- **Conversion:** Relevant scan CTA from case/About.
- **Fallback:** Contact/email with a precise credibility/procurement question.
- **Required proof:** Claim register, permissions, real case artifacts, verified identity/credentials and method artifacts. Lab/concept proof alone must be perceived as such.

## Journey E — Uncertain visitor

- **Trigger/landing:** Organization is slow but cause might be process, software, data or integration; lands on homepage or scan.
- **Questions:** Which route applies? Must I diagnose it myself? What information is safe to share? Will the scan choose a tool too early?
- **Objections:** Route labels feel technical; fear of choosing wrong; form asks details they do not know; combined sounds expensive/unbounded.
- **Content consumed:** Conflict → Hidden Drag → Diagnosis → Two Routes `onzeker/gecombineerd` → scan route explanation/common layers → intake.
- **Decision points:** Can they name visible symptoms? If no clear root cause, select `unsure`; if both are visibly coupled, select `combined`; understand route may be corrected after intake.
- **Likely exits:** Return to service comparison, Method, email or abandon at required technical fields.
- **Conversion:** `Ik weet het nog niet` → shared intake with symptom-first prompts and optional affected-system detail.
- **Fallback:** Verkennend gesprek/email; no penalty for uncertainty.
- **Required proof:** Scan report anatomy showing observations versus assumptions, combined current-state map and example of a recommendation not to build.

## Cross-journey prototype states

| State | Required behavior |
|---|---|
| Route selected | Clearly show selection and allow change without losing shared fields |
| Proof unavailable | Explain that evidence is being prepared; never substitute invented metrics |
| Commercial fact unresolved | Omit or mark internally in prototype, never expose vague public promise |
| Form recoverable error | Retain safe fields, focus error summary, identify exact corrections |
| Form unavailable | Offer email and preserve privacy warning; do not imply submission succeeded |
| Success | Confirm receipt only after server success; state next step without unsupported timing |

## Journey test measures

For each participant record route chosen, first CTA perceived, confidence (1–5), incorrect assumptions, evidence requested and whether they can explain the scan. Critical failure: selecting a route feels irreversible, AI is perceived as the product, the site appears to be a multi-person agency, or a lab/concept is mistaken for client proof.
