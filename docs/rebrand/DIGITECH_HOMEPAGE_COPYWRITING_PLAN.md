# Digitech Solutions — Copywriting Plan & Codex Handoff

**Datum:** 22 juli 2026
**Bron:** actuele rebrandprototype, contentdata en strategische documentatie uit de copywriting-handoff
**Doel:** een complete Nederlandse copylaag leveren die Codex gecontroleerd in het bestaande prototype kan verwerken.

---

## 1. Executive decision

De huidige aanpak is goed, met één belangrijke aanpassing:

> Gebruik Codex niet als primaire copywriter. Gebruik Codex als implementatie-agent voor een vooraf vastgelegd copy contract.

De aanbevolen workflow is:

1. bestaande prototypecomponenten en beschikbare ruimte als vaste structuur gebruiken;
2. copy per component vastleggen in dit document;
3. iedere tekst classificeren als `DEFINITIEF`, `VOORLOPIG`, `BEWIJS NODIG` of `GOEDKEURING NODIG`;
4. Codex alleen laten vervangen, structureren en testen;
5. daarna visueel controleren op desktop en mobiel;
6. pas vervolgens microcopy, scanformulier en onderliggende pagina’s afronden.

De vier verhaalfases blijven leidend:

> **Chaos → Bezinning → Arbeid → Resultaat**

De bestaande technische hoofdstukken blijven daarbinnen bruikbaar:

- Chaos: hero en verborgen frictie;
- Bezinning: diagnose en routehypothese;
- Arbeid: expertise-routes, methode en founder-led uitvoering;
- Resultaat: scan-output, beheersbare vervolgstap en conversie.

**Artifact label**

# 7. Content architecture recommendation

## Houd copy buiten `RebrandPage.tsx`

De huidige implementatie bevat veel tekst direct in JSX. Verplaats de homepagecopy naar één typed contentbestand, bijvoorbeeld:

`lib/rebrand/homepage-copy.ts`

Aanbevolen structuur:

```ts
export const homepageCopy = {
  navigation: {},
  hero: {},
  hiddenDrag: {},
  diagnosis: {},
  routes: {},
  method: {},
  founder: {},
  result: {},
  scan: {},
  footer: {},
  metadata: {},
} as const;
```

Voordelen:

- copy kan worden aangepast zonder de componentlogica te wijzigen;
- desktop- en mobiele varianten zijn expliciet;
- claims en placeholders zijn beter auditeerbaar;
- tests kunnen verplichte labels en verboden marketingzinnen bewaken;
- Codex hoeft niet verspreid door grote JSX-bestanden te herschrijven.

## Voeg statustags niet toe aan publieke UI

`DEFINITIEF`, `VOORLOPIG`, `BEWIJS NODIG` en `GOEDKEURING NODIG` zijn redactionele metadata. Toon deze niet als normale websitecopy.

Gebruik intern bijvoorbeeld:

```ts
type CopyStatus =
  | "definitief"
  | "voorlopig"
  | "bewijs-nodig"
  | "goedkeuring-nodig";
```

Voor productie kan Codex alleen `definitief` en expliciet toegestane `voorlopig`-content renderen.

# 8. Implementatievolgorde voor Codex

## Fase A — content refactor

1. Maak `lib/rebrand/homepage-copy.ts`.
2. Verplaats bestaande copy uit `RebrandPage.tsx` en `content.ts`.
3. Voeg de nieuwe copy toe zonder layout of motion te veranderen.
4. Behoud anchors, heading IDs en componentstructuur.
5. Voeg de ontbrekende resultaatsectie alleen toe wanneer dit zonder redesign in de huidige storyflow past.

## Fase B — copy fit review

Controleer:

- regels en wrapping op 1440, 1024, 768, 375 en 320 px;
- headings maximaal circa 2–4 regels op mobiel;
- buttons niet onnodig over 3 regels;
- geen horizontale overflow;
- details/accordions blijven leesbaar;
- motiontriggering verandert niet door gewijzigde contenthoogte;
- CTA’s blijven logisch;
- scan blijft eerlijk prototypegedrag tonen.

## Fase C — claim and placeholder audit

Controleer dat:

- geen ervaringsjaar wordt gepubliceerd;
- geen certificaat wordt gepubliceerd;
- geen echte case wordt geïmpliceerd;
- geen exacte Scan-belofte wordt gedaan;
- founder-led qualifier behouden blijft;
- AI alleen als ondersteunende werkwijze wordt genoemd;
- geen verboden marketingfrases terugkomen.

## Fase D — tests

Breid `lib/rebrand/content.test.ts` uit met:

- verboden marketingzinnen;
- één dominante CTA per hoofdscene;
- verplichte founder qualifier;
- verboden cijfers/percentages in onbewezen copy;
- geen succesclaim voor prototype-intake;
- correcte vieraktenvolgorde;
- verplichte sensitive-data warning zodra intake wordt gebouwd.

## Resultaatsectie

Controleer of de huidige prototypeflow al expliciet eindigt met Resultaat.

Wanneer die inhoudelijk ontbreekt, voeg tussen founder-led en Scan een compacte Resultaatsectie toe met:

- kicker: De beoogde eindtoestand
- headline: Niet alleen gemoderniseerd, maar weer onder controle.
- drie blokken: Zichtbare flow, Aanpasbare systemen, Belegd eigenaarschap
- een eerlijk gelabelde proof/report-anatomy placeholder

Maak hiervan geen testimonial- of statistieksectie.

Wanneer toevoegen een grote layout- of motionwijziging vereist, implementeer de copydata wel maar rapporteer de sectie als deferred in plaats van de architectuur zelfstandig te verbouwen.

## Founderbio

Gebruik alleen de interne placeholder uit het copyplan.

Toon geen interne redactiestatus als normale marketingcopy.

Wanneer de UI een publieke placeholder nodig heeft, gebruik:

“Meer over Subor en zijn relevante ervaring volgt na inhoudelijke verificatie.”

Bij voorkeur wordt deze placeholder alleen op de prototype-route getoond en niet op productie.

## Modernization Scan

Behoud prototype-eerlijkheid.

- Geen formulierdata verzenden.
- Geen success state.
- Geen prijs, duur of deliverable beloven.
- Behoud werkende fallback naar een verkennend gesprek.
- Gebruik de copy uit het contract.
- Zet sensitive-data microcopy pas bij een echte intake.

## Metadata

Pas alleen prototype-route metadata aan wanneer dat veilig kan zonder canonical of productie-SEO te wijzigen.

Behoud noindex.

## Tests en checks

Voer uit:

- git diff --check
- npx tsc --noEmit
- npm run build
- bestaande tests

Controleer browserweergave op:

- 1440
- 1024
- 768
- 375
- 320 px

Controleer:

- geen horizontale overflow;
- geen heading overflow;
- CTA-labels passen;
- details blijven bruikbaar;
- copyhoogte breekt motion niet;
- reduced motion blijft werken;
- geen dode links;
- geen onbewezen claim;
- prototype-intake blijft eerlijk.
