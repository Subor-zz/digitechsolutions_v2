# Low-Fidelity Page Specifications

## Shared shell and density scale

All pages use a skip link, header, one `main`, contextual breadcrumb where useful, footer, visible focus and a single dominant CTA. **Low density** ≈ 3–5 short sections; **medium** ≈ 6–8; **high** ≈ 8–11 but must remain scannable. Desktop layout may use 12 columns; mobile is one semantic column at 320 px. Primary copy and conversion never require JavaScript.

## 1. Homepage

- **Goal/audience/intent:** Orient operations first, then IT/founder evaluators arriving by brand, referral or broad problem search.
- **Question:** “Does Digitech understand what slows us down, and what is the safe first step?”
- **CTA:** `Start met een Modernization Scan`; secondary `Bekijk de werkwijze`.
- **Hierarchy/order/density:** Nine locked story chapters; medium-high but short modules. Detail in `docs/rebrand/HOMEPAGE_WIREFRAME_SPEC.md`.
- **Trust/proof:** Diagnosis layers, delivery gates, classified cases, founder identity and explicit limits. Requires two cases, verified bio and scan facts.
- **Mobile/accessibility:** Same chapter order; no pinning requirement; labelled static diagrams; sticky CTA must not obstruct content; headings remain sequential.
- **SEO:** Brand/category hub targeting workflow- and application-modernisering naturally; Organization/Person facts only when verified.
- **Dependencies/exits:** Case and scan content; exits to both services, Method, cases, About and scan.

## 2. Workflow modernization

- **Goal/audience/intent:** Help operations buyers recognize a suitable workflow problem while satisfying IT/data reviewers.
- **Question:** “Can this process be improved without losing exceptions, control or ownership?”
- **CTA:** `Start een workflow-led scan`; secondary `Bekijk de werkwijze` or verified workflow case.
- **Hierarchy/order/density:** Hero + fit signals → hidden-friction example → current-state mapping → simplify/integrate/automate options → human controls/data/integrations → expected outputs → non-fit/limits → proof → FAQ → CTA; high.
- **Trust/proof:** Before/after process map labelled illustrative, exception handling, audit trail, ownership, verified workflow case. Never promise full automation or savings.
- **Mobile/accessibility:** Before/after stacked, not horizontal-only; tables become labelled lists; diagrams have text equivalents.
- **SEO:** Primary service landing page; unique copy stronger than homepage; Service schema only with approved facts.
- **Dependencies/exits:** Workflow example, case and FAQs; exits to scan, application route when software is root cause, case and contact.

## 3. Application modernization

- **Goal/audience/intent:** Give IT and founder/product visitors a controlled alternative to a blind rewrite.
- **Question:** “How will legacy risk be assessed and reduced without destabilizing the application?”
- **CTA:** `Start een application-led scan`; secondary `Bekijk de methode` or verified application case.
- **Hierarchy/order/density:** Hero + warning signs → application layers → assessment evidence → option matrix (secure/refactor/replatform/replace/rebuild/retire) → phased migration/rollback/testing → outputs → limits/non-fit → proof → FAQ → CTA; high.
- **Trust/proof:** Dependency/sample architecture evidence, tests, authorization/data/integration review boundaries, rollback and exclusions. No security guarantee or instant rewrite.
- **Mobile/accessibility:** Layer view becomes ordered sections; option matrix becomes cards; code/interface imagery is supplementary.
- **SEO:** Primary application-modernization service page; connects migrated technical-debt insights.
- **Dependencies/exits:** Application case and technical artifacts; exits to scan, workflow route, Method, insight and contact.

## 4. Modernization Scan

- **Goal/audience/intent:** Turn uncertainty into a qualified, low-risk first step without overclaiming commercial detail.
- **Question:** “Is the scan suitable, what route applies, what do I provide and receive?”
- **CTA:** `Start de intake`; secondary `Plan een verkennend gesprek`/email.
- **Hierarchy/order/density:** Promise → four route choices → fit/non-fit → common layers → outputs → process → practical facts/placeholders → standalone value/limits → FAQ → intake; high.
- **Trust/proof:** Sample report anatomy, evidence/assumption distinction, exclusions and privacy warning. Requires confirmed duration/input/scope before final design.
- **Mobile/accessibility:** Route cards stack; step indicator uses text; errors summarized and inline; no auto-scroll that steals focus.
- **SEO:** Primary conversion/service page; Service structured data only after price/duration facts are stable.
- **Dependencies/exits:** Product facts, form/legal providers; exits to intake, services, email/contact, privacy.

## 5. Method

- **Goal/audience/intent:** Let technical and procurement validators inspect how founder-led delivery is controlled.
- **Question:** “How do scope, quality, security and handover stay visible?”
- **CTA:** `Start met een scan`; secondary verified case.
- **Hierarchy/order/density:** Principle → Scope/Map/Simplify/Build/Verify/Transfer → artifacts per step → roles/decision rights → quality gates → risk/escalation → handover → case → CTA; medium-high.
- **Trust/proof:** Sample decision log, acceptance criteria, test/security boundary and transfer checklist; avoid empty badge claims.
- **Mobile/accessibility:** Vertical ordered list; timeline connector decorative; step links remain usable without motion.
- **SEO:** Supports branded methodology and internal linking, not a keyword-stuffed service page.
- **Dependencies/exits:** Method artifacts and verified claims; exits to case, About, services and scan.

## 6. Cases overview

- **Goal/audience/intent:** Help returning/referral visitors find relevant, inspectable proof.
- **Question:** “Is there evidence comparable to my situation, and what kind is it?”
- **CTA:** `Bekijk de case`; secondary `Start met een scan`.
- **Hierarchy/order/density:** Evidence policy intro → workflow group → application group → optional lab group → scan CTA; low-medium. No filters until volume requires them.
- **Trust/proof:** Classification label, permission-safe context, evidence type, result qualifier and last review date on each card.
- **Mobile/accessibility:** One-column cards; classification is text, not color; no carousel.
- **SEO:** Index page for case discovery; ItemList only for published cases.
- **Dependencies/exits:** Two approved cases; if none, do not publish a hollow index. Exits to details, services and scan.

## 7. Case detail

- **Goal/audience/intent:** Demonstrate reasoning and evidence rather than a polished success story alone.
- **Question:** “What was observed, decided, delivered, measured and deliberately excluded?”
- **CTA:** Relevant scan route; secondary related insight/service.
- **Hierarchy/order/density:** Classification/disclosure → summary → context/problem/friction → evidence → analysis/decision → scope → solution/map → quality gates → result → limitations/not built → next step; high.
- **Trust/proof:** Evidence citations, permission state, baseline/method/timeframe for metrics, anonymization disclosure and limitations.
- **Mobile/accessibility:** Evidence table becomes labelled entries; images zoom without trapping; all diagrams have captions/text equivalents.
- **SEO:** Long-form proof page; no unsupported Review schema; canonical and classification reflected in metadata.
- **Dependencies/exits:** Completed `docs/rebrand/CASE_TEMPLATE.md` and approvals; exits to scan, service, cases and insight.

## 8. About Subor

- **Goal/audience/intent:** Resolve accountability, capability, company-size and scope questions.
- **Question:** “Who will I work with, what can he credibly own, and where are the boundaries?”
- **CTA:** `Start met een scan`; secondary `Bekijk de werkwijze`.
- **Hierarchy/order/density:** Direct identity → short bio/photo → relevant chronology → capability-to-outcome map → working style → founder-led benefits/limits → transparent specialist model → verified credentials → CTA; medium.
- **Trust/proof:** Real photo, sourced dates/credentials, clear first-person voice and no implied internal team.
- **Mobile/accessibility:** Photo follows heading/summary; chronology is semantic list; credentials are linked text.
- **SEO:** Person/brand trust page; Person schema facts must match register.
- **Dependencies/exits:** Bio, photography, claim verification; exits to Method, cases, scan and contact.

## 9. Insights overview

- **Goal/audience/intent:** Help search/research visitors learn and move toward a relevant route.
- **Question:** “Which practical modernization guidance applies to my problem?”
- **CTA:** `Lees het inzicht`; secondary contextual service/scan.
- **Hierarchy/order/density:** Editorial promise → featured item → workflow/application/decision topics → recent items → scan CTA; medium.
- **Trust/proof:** Author, publish/update date, topic and source standard; no fake popularity or empty categories.
- **Mobile/accessibility:** One-column article list; topic links wrap; no horizontal scroller.
- **SEO:** Indexable editorial hub replacing `/blog`; pagination only when required.
- **Dependencies/exits:** Approved migrated articles; exits to detail, service, case and scan.

## 10. Insight detail

- **Goal/audience/intent:** Answer a specific search/research question with a practical, qualified framework.
- **Question:** Defined by article title; reader must understand limits and next decision.
- **CTA:** Contextual relevant service/scan; secondary related case/insight.
- **Hierarchy/order/density:** Breadcrumb → title/dek/author/date → key takeaway → article sections → sources/caveats → author note → related content → CTA; content-dependent high.
- **Trust/proof:** Primary/authoritative sources, last-reviewed date, disclosed illustrative examples and claim-register compliance.
- **Mobile/accessibility:** Comfortable measure, semantic tables/captions, no sticky share rail; headings support navigation.
- **SEO:** Article schema, canonical, unique metadata and internal links; preserve strong migrated intent with direct redirects later.
- **Dependencies/exits:** Editorial review and source verification; exits to service, scan, case and other insight.

## 11. Contact

- **Goal/audience/intent:** Serve ready buyers and enquiries that do not fit the scan flow.
- **Question:** “How can I safely contact Subor and what happens next?”
- **CTA:** `Verstuur bericht`; secondary email and scan link.
- **Hierarchy/order/density:** Expectation → compact general form → privacy/data warning → direct email/contact facts → scan signpost; low.
- **Trust/proof:** Real contact facts, non-promissory response expectation and privacy link. Do not ask budget unless a justified need emerges.
- **Mobile/accessibility:** One column, correct input types/autocomplete, visible required status, error summary/focus and persistent alternative email.
- **SEO:** Contact/LocalBusiness facts only when verified; prevent form-success indexing.
- **Dependencies/exits:** Form provider/legal copy; exits to success/error recovery, privacy, scan and email.

## 12. Privacy

- **Goal/audience/intent:** Explain actual processing and rights in plain Dutch.
- **Question:** “Which data is used, why, by whom, for how long, and how do I exercise rights?”
- **CTA:** Privacy contact; secondary return to previous task/home.
- **Hierarchy/order/density:** Controller/contact → data/purpose/basis → forms → analytics/cookies → processors/transfers → retention → rights/complaints → security → changes; high legal content with contents list.
- **Trust/proof:** Actual provider/DPA/retention facts and version/effective date; legal review.
- **Mobile/accessibility:** Semantic contents links, readable measure, no legal modal-only presentation.
- **SEO:** Indexable legal transparency page; no marketing structured data.
- **Dependencies/exits:** Provider decisions and legal owner; exits to contact/home/terms.

## 13. Terms

- **Goal/audience/intent:** Provide applicable commercial/legal framework without substituting proposal-specific terms.
- **Question:** “Which entity and conditions govern a possible engagement?”
- **CTA:** Contact with questions; secondary scan/home.
- **Hierarchy/order/density:** Entity/application/version → proposals/scope → fees/payment → client responsibilities/access → IP/confidentiality/data → warranty/liability → termination/force majeure/law; high legal content with contents list.
- **Trust/proof:** Legal review, entity identifiers and version history.
- **Mobile/accessibility:** Same as Privacy; tables avoided where prose/list works.
- **SEO:** Legal page, indexability decision by SEO/legal; canonical required.
- **Dependencies/exits:** Approved legal copy; exits to contact/privacy/home.

## 14. 404

- **Goal/audience/intent:** Recover visitors from old links or typing errors without pretending success.
- **Question:** “Where can I continue?”
- **CTA:** `Ga naar de homepage`; secondary scan, services, cases or insights.
- **Hierarchy/order/density:** Clear `Pagina niet gevonden` → one-sentence explanation → four useful destinations → contact fallback; low.
- **Trust/proof:** None; never auto-redirect or return HTTP 200.
- **Mobile/accessibility:** Focus starts at main heading after navigation; plain links; no animated error spectacle.
- **SEO:** Real 404 status, noindex; track path only after privacy review and strip query/PII.
- **Dependencies/exits:** Final navigation/routes; all exits explicit.

## 15. Form success and error states

- **Goal/audience/intent:** Confirm outcome unambiguously and preserve recovery after submission.
- **Question:** “Was my information received, what happens next, and what can I do if it failed?”
- **CTA:** Success: `Bekijk de werkwijze` or home; error: `Probeer opnieuw`. Secondary email.
- **Hierarchy/order/density:** Status heading → submission outcome → non-promissory next step → privacy/reminder not to email secrets → next links; low.
- **Trust/proof:** Reference ID only if non-sensitive and actually supported; no claim of receipt before server confirmation.
- **Mobile/accessibility:** Move focus to status heading/error summary; `role=status` for success and `role=alert` for blocking errors; retain non-sensitive entered values on recoverable error; no toast-only state.
- **SEO:** Dedicated success URLs are noindex and must not expose form data; errors normally remain on form route.
- **Dependencies/exits:** Backend/provider behavior and response promise; exits to retry, email, scan/Method/home.
