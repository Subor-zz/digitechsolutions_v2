## Blockers

- Application source and Git history are now available. Vercel account roles/settings and live analytics still require owner/network verification.
- No two approved, evidence-labelled launch cases are present.
- Public metrics, experience and certification claims are unverified and internally inconsistent.
- Modernization Scan remains commercially and operationally unbounded in several acceptance criteria.
- Form/analytics processors, retention and legal basis are unknown.

These do not prevent low-fidelity layout exploration if placeholders are visibly marked; they block final proof design, production initialization or launch as indicated.

## Risks

- **P1 credibility/data integrity:** migrating unsupported metrics or fabricated case inference would undermine the new evidence-led position.
- **P1 privacy/security:** reusing an unknown form/analytics path could expose contact PII or create inaccurate legal copy.
- **P1 delivery:** building before Vercel role/rollback verification and migration boundaries are approved risks deployment or rollback failure.
- **P1 SEO:** blanket redirects or migration without Search Console/backlink evidence could lose qualified traffic.
- **P2 scope:** nine homepage chapters can become long if service detail and proof are not aggressively edited.
- **P2 performance/accessibility:** scroll motion can violate mobile/reduced-motion/Core Web Vitals without the prescribed static-first spike.
- **P2 brand:** blue/orange and secondary grays need a measured contrast matrix; current values are roles, not approved text pairings.

## Assumptions

The live domain observed on the audit date represents the current public site; the supplied folder is intentionally a planning package; Dutch remains launch language; Digitech remains founder-led; the scan can be wireframed with labelled unknown commercial facts; and no current route is deleted or redirected during this sprint.

## Recommended next phase

Run a **low-fidelity UX and evidence-definition sprint**, in parallel with owner-side proof gathering but without production code. The sprint answers hierarchy, navigation, comprehension, route selection, scan intake and mobile story compression. Motion is represented only as start/end annotations.

1. Desktop and 320 px wireframes for all launch page types: homepage, both services, scan, method, cases/index/detail, about, insights/index/detail, contact and legal template.
2. Clickable prototype for the five journeys in `docs/rebrand/SITEMAP_AND_USER_JOURNEYS.md`.
3. Homepage chapter-length budget and copy-fit pass; mark case/scan unknowns as content dependencies.
4. Scan triage/intake flow including workflow, application, combined/unsure, validation, privacy warning, error and success states.
5. Global navigation, mobile menu, breadcrumb, footer, sticky CTA and focus-order annotations.
6. Static start/end diagrams for hero, Hidden Drag, diagnosis, split and method; mobile and reduced-motion frames.
7. Case-detail evidence template with classification, source/permission state, result context and exclusions.
8. Content model/component inventory mapped to each page block.
9. Five-person comprehension/usability test (operations, IT and founder mix) with 5-second/30-second questions and critical-task results.
10. Decision review updating scan placeholders, proof blockers, IA issues and visual-spike brief.

## Exit criteria for the next phase

Users can identify both offers, scan and founder-led model; unsure visitors choose the combined path; all primary journeys work at 320 px and keyboard order is specified; no page depends on motion; proof placeholders cannot be mistaken for real claims; and remaining content decisions have owners before visual design.

## Phase 0 status statement

Do not label the entire phase “complete” yet. Label it: **“Specification complete; gate conditional for low-fidelity UX.”** Move to visual design only after scan boundaries, two cases and the claim audit are approved. Move to engineering only after low-fidelity/visual-motion gates pass and Vercel role/rollback access is verified; repository/source availability is no longer a blocker.
